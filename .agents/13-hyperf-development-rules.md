# Hyperf 开发执行规则

本文档用于约束本项目后续所有后端代码分析、设计、修改、生成和测试工作。处理代码任务前，必须先进入当前项目上下文，优先理解并遵循项目已有结构、命名规范、目录分层、封装方式、响应格式、异常处理方式、数据库模型、配置方式和业务约定。

## 1. 优先级

### 1.1 第一优先级：当前项目已有实现

如果项目中已经存在类似功能、工具类、Logic、Model、Controller、DTO、枚举、异常类、响应封装、验证规则、中间件、事件、队列、定时任务或配置方式，必须优先复用项目已有写法，不得重新创造一套实现。

### 1.2 第二优先级：Hyperf 3.1 官方方式

如果项目中没有现成实现，优先使用 Hyperf 3.1 官方推荐方式和官方组件完成，包括但不限于：

- Controller
- Request / FormRequest
- Validation
- Middleware
- ExceptionHandler
- DI 容器
- Config
- Event / Listener
- Model
- Query Builder
- Migration
- Logger
- Cache
- Redis
- Async Queue
- Crontab
- Command
- Context
- Coroutine 相关能力

### 1.3 第三优先级：成熟通用实践

只有在项目已有实现和 Hyperf 官方能力都不能很好满足需求时，才考虑使用成熟 PHP 通用实践或自定义封装。

## 2. 禁止事项

- 不要默认使用普通 PHP 脚本式写法。
- 不要直接照搬 TP6 写法；Hyperf 和 TP6 有相似处，但必须优先确认 Hyperf 3.1 官方写法。
- 不要使用 `echo`、`exit`、`die`、`var_dump` 处理业务流程。
- 不要随意 `new` 业务依赖，优先使用项目已有单例方式或 Hyperf DI。
- 不要硬编码配置，配置应放到 `config/autoload/*.php` 或项目已有配置结构中。
- 不要随意新增和项目风格不一致的工具类、基础类或封装。
- 不要编造 Hyperf 不存在的类、方法、命令或配置。
- 不要绕过项目统一响应格式、异常处理和数据库模型约定。

## 3. 写代码前必须判断

每次生成或修改代码前，必须先判断：

- 项目里是否已有类似代码可以参考。
- 是否已有统一响应格式。
- 是否已有业务异常类或错误码封装。
- 是否已有参数校验方式。
- 是否已有 Controller / Logic / Model 分层约定。
- 是否已有配置文件、常量或枚举。
- 是否需要使用 Hyperf 官方组件。
- 是否存在协程安全问题。
- 是否需要事务、日志、队列、缓存或事件。
- 是否会影响现有项目风格和已有接口行为。

## 4. 实现要求

- 优先复用当前项目已有 Controller / Logic / Model 分层。
- Controller 只负责请求参数读取、调用 Logic、返回统一响应。
- Logic 负责业务规则、第三方调用、事务边界和数据组装。
- Model 负责表名、字段映射、常量、类型转换和查询能力。
- 响应优先使用项目已有 `JsonTable`。
- 数据库操作优先使用项目已有 Model 或 Hyperf Query Builder。
- 参数校验优先使用项目已有校验方式；没有时再考虑 Hyperf Validation / FormRequest。
- 异常处理优先沿用项目已有 `JsonTable::withError`、业务错误码或异常机制。
- 日志优先使用 Hyperf Logger 或项目已有日志封装。
- 请求级数据优先使用 Hyperf Context 或当前请求对象，避免协程污染。
- 涉及第三方 SDK 时，优先复用已有 SDK facade、driver 和配置转换方法。

## 5. 新增封装前必须说明

如果确实需要新增封装，必须先说明：

- 为什么项目已有能力不够。
- 为什么 Hyperf 官方能力不够。
- 新封装解决什么问题。
- 是否会影响现有结构。
- 如何保持可维护性。

## 6. 回答结构

涉及代码设计、功能实现、问题分析时，默认按以下结构输出：

1. 推荐方案
2. 为什么这样符合当前项目 / Hyperf 官方方式
3. 涉及的文件
4. 需要执行的命令
5. 关键代码
6. 注意事项

如果是简单修复或已经直接完成的任务，可以简化，但必须保留关键结论、涉及文件和验证结果。

## 7. 总目标

像真正接手当前 Hyperf 项目的后端工程师一样工作：先看项目，优先复用项目已有写法；项目没有时，再使用 Hyperf 官方方式；最后才考虑自定义实现。不要只给“能跑”的代码，要给适合当前项目长期维护的代码。
