# sql/AGENTS.md

本目录维护 `cyberpal_sweatshop` 项目的数据库初始化脚本、字典、菜单、功能、基础数据和历史权限参考。处理本目录时，以数据定义仓库标准维护 SQL，不把它当作应用源码目录。

修改 SQL 文件前必须先阅读本文件，并结合仓库根目录 `AGENTS.md` 执行。

## 目录结构

- `.agents/`：`sql/AGENTS.md` 可引用的索引关联文件、SQL 模板、补充规范和检查清单目录。该目录按需创建，并纳入仓库管理。
- `initialization/`：正式初始化脚本目录，按文件名前缀排序执行，是当前维护数据库结构和基础数据的主目录。
- `permission/`：暂时弃用的历史权限 SQL。后续处理不要新增、修改、删除或同步该目录文件。
- `privilege/`：暂时弃用的历史 CSV 参考源。后续处理不要新增、修改、删除或同步该目录文件。

新增 SQL 资产时，应放在最接近用途的现有目录中。正式初始化和种子数据优先维护 `initialization/`，临时执行脚本不应直接混入正式初始化链路。

修改正式 SQL 内容时，除更新对应正式脚本外，还需要将本次任务相关的可执行更新 SQL 同步生成到仓库根目录 `.tmp/` 下。`.tmp/` 中的更新脚本应只包含本次变更所需的增量 SQL，文件名需体现日期和用途，便于上线或手工补丁执行。

历史目录映射：

- 旧规范中的 `data-initialization/` 已收敛为当前 `initialization/` 目录。
- 旧规范中的 `1-base`、`2-***`、`3-code`、`10-data`、`11-dict` 分别对应当前 `01-base.sql`、`02-cyberpal_sweatshop.sql`、`03-code.sql`、`10-data.sql`、`11-dict.sql`。
- 旧规范中的 `20` 以上码表单表文件对应当前 `21-`、`22-`、`23-` 等 `codedata_<table>.sql` 静态码表文件。
- 旧规范中的 `data-permission/` 对应当前历史参考目录 `permission/` 和 `privilege/`，当前均暂时弃用。
- 旧规范中的 `data/`、`data-release/`、`data-temp/`、`data-static/` 在当前正式初始化链路中没有直接维护目录；如需新增同类临时、上线或统计 SQL，应先确认用途和归档位置，不要默认混入 `initialization/`。

历史规则中允许使用中文文件名记录初始化数据、手动查询、上线执行、开发分支临时或周期统计 SQL。当前仓库没有为这些用途保留独立正式目录，后续如需恢复，应在文件名或相邻说明中写清使用方式、执行场景、所属分支或版本，并在执行完成后按年度或项目约定归档。上线文件历史命名参考为 `[release-版本-时间]`，开发临时文件历史命名参考为 `[时间-story**-开发人]`。

## 初始化脚本顺序

`initialization/` 按文件名前缀排序执行。维护时需要保留现有顺序和文件职责：

- `01-base.sql`：基础或框架表，例如 `cs_dict`、`cs_menu`、`cs_function`、`cs_user`、角色、权限、文件、服务商和系统记录等。
- `02-cyberpal_sweatshop.sql`：本项目业务表。
- `03-code.sql`：码表定义和小规模码表初始化数据。
- `10-data.sql`：项目运行所需基础数据，例如初始化用户、角色和关系等。
- `11-dict.sql`：字典元数据。该文件有特殊格式要求，不要自动格式化或整文件生成。
- `12-menu.sql`：各端菜单初始化数据。
- `13-function.sql`：菜单功能按钮与动作初始化数据。
- `21-codedata_housearea.sql`、`22-codedata_country.sql`、`23-codedata_nation.sql`：行政区划、国家、民族等大规模静态码表数据。
- `24-permission.sql`：初始化权限关系数据。

修改业务表时，直接更新对应初始化文件中的 `CREATE TABLE` 语句。字段注释、索引、默认值、软删除字段等需要与邻近表保持一致。

## 弃用目录

`permission/` 和 `privilege/` 暂时弃用。

- 不要在 `permission/` 中新增、修改、删除或同步菜单、功能、角色授权、用户授权 SQL。
- 不要在 `privilege/` 中新增、修改、删除或同步 CSV 参考源文件。
- 需要调整菜单或初始化权限相关数据时，优先维护 `initialization/` 下对应初始化脚本，除非用户明确要求恢复或处理这两个弃用目录。

## SQL 风格

- 表名使用 `cs_` 前缀。
- 新建或按任务要求调整字段前缀时，使用物理表名去掉 `cs_` 后的 snake_case 各单词首字母组合并追加下划线，例如 `cs_dict_drug -> dd_`、`cs_drug_storage_method -> dsm_`。历史表若已形成既有前缀，按任务要求最小化修改。
- 时间字段使用 `create_time`、`update_time`、`delete_time` 语义名并保留字段前缀，例如 `dd_create_time`、`dd_update_time`、`dd_delete_time`；字段类型使用 `BIGINT`，默认值使用 `0`。
- 业务表之间关联其他业务表主键字段时，一般使用 `INT` 保存目标表主键 ID；字段命名使用"当前表字段前缀 + 关联表去掉 `cs_` 后的 snake_case 表名"，不追加目标主键字段名。例如 `cs_dict_drug_tracing_code` 关联 `cs_dict_drug.dd_id` 时字段命名为 `ddtc_dict_drug`，不命名为 `ddtc_drug_id` 或 `ddtc_dict_drug_id`。
- 业务表之间关联其他业务表非主键字段时，字段命名使用"当前表字段前缀 + 关联表去掉 `cs_` 后的 snake_case 表名 + 目标字段去掉目标表前缀后的语义名"。例如关联 `cs_dict_drug.dd_code` 时字段命名为 `ddtc_dict_drug_code`。
- 只有关联码表类数据时使用对应 `code` 字段，并按上述非主键字段规则保留目标字段语义。
- 数字枚举状态字段使用短表前缀加 `state`，例如 `xxx_state`，字段类型使用 `TINYINT`。
- 状态码类字段使用短表前缀加 `status`，例如 `xxx_status`，字段类型使用字符串。
- 初始化脚本中创建表前使用显式 `DROP TABLE IF EXISTS`，保持与现有文件一致。
- 保留软删除约定，例如 `*_delete_time`，以及状态和删除时间相关索引。
- SQL 语句风格以正在编辑的文件为准；历史文件中大小写 SQL 关键字混用，不要做无关统一格式化。新增语句可在不破坏上下文风格的前提下优先使用大写 SQL 关键字。
- 文件名应描述用途，并保留仓库现有的数字排序模式。

## 字典规则

维护 `initialization/11-dict.sql` 时：

- 不要自动格式化 `initialization/11-dict.sql`。
- 不要用工具整文件重新生成字典 SQL。
- 字典只描述有调整的字段，不需要管理未变更的默认值。
- `d_tablename` 为表名去掉前缀下划线后的首字母大写驼峰形式。
- `di_fieldname` 为完整字段名，包含与物理表一致的字段前缀。
- `di_name` 为页面展示字段名，应在当前字典上下文内精简，不要重复表名或字典名称已表达的信息。码表类字典中主键字段通常命名为 `序号`，编码字段命名为 `编码`，名称字段命名为 `名称`。
- 字段前缀默认按物理表名去掉 `cs_` 后的 snake_case 各单词首字母组合并追加下划线。历史表若已沿用其他前缀，除非任务明确要求，不做无关重命名。
- `100` 以下字典 ID 不可用。
- `100-300` 为码表字典。
- `500-1000` 为框架所需表字典。
- `1000` 以上为业务系统字典。
- 如果同一张物理表需要不同应用侧设计，应创建独立字典项，ID 按现有编号规范延续。

### 字典项字段配置

生成或维护 `cs_dict_item` 时，需要同时考虑后端通用 CRUD 逻辑和前端页面生成逻辑。`di_fieldname` 在 SQL 中保留完整物理字段名；前端读取字典定义时会去掉当前表的 `d_prefix`；新增、修改、查询条件进入后端时会按 `d_prefix` 补回物理字段前缀。

`di_required`、`di_readonly`、`di_inputed`、`di_curd` 均使用位值组合：

- `1`：刷新/列表。
- `2`：新增。
- `4`：修改。
- `8`：读取/详情。
- `16`：删除。
- 可相加组合，例如 `6=新增+修改`，`9=列表+详情`，`14=新增+修改+详情`，`15=列表+新增+修改+详情`。

字段控制规则：

- `di_curd` 是后端强控制字段，决定该字段是否参与通用 CRUD。常用值：`15` 表示列表、新增、修改、详情都参与；`9` 表示只参与列表和详情；`0` 表示不参与通用 CRUD。
- `di_required` 是后端新增/修改必填校验字段，只对 `2`、`4` 生效。常用值：`2` 表示新增必填，`4` 表示修改必填，`6` 表示新增和修改都必填。
- `di_readonly` 是后端新增/修改只读字段，只对 `2`、`4` 生效。自增主键通常设置 `di_readonly=6`。
- `di_inputed` 主要是前端页面展示元数据，不替代后端 `di_curd`、`di_required`、`di_readonly`。常用值：`15`、`14`、`9`、`8`、`0`。
- `di_select` 是前端枚举、下拉、单选或状态标签配置，格式为 `值-显示名;值-显示名`，例如 `0-停用;1-正常`。
- `di_default` 是默认值文本，若字段配置了 `di_select`，默认值应优先取 `di_select` 中已声明的值。
- `di_filtered=1` 表示该字段适合作为前端筛选条件；若同时配置 `di_select`，前端可生成下拉筛选。
- `di_key_dict`、`di_key_table`、`di_key_field`、`di_key_show` 用于配置关联字典或关联表展示。默认不要设置 `di_key_join_name`，后端会按 `di_key_table` 生成关联别名；只有同一个字典中有两个及以上字段关联到同一张表时，才为第二个及后续字段手动设置 `di_key_join_name`，避免 SQL 联表别名冲突。

新增字典项时优先按上述语义填写。维护历史字典时，不要为了统一 `di_required`、`di_inputed`、`di_curd` 等值而重写无关行；只调整本次任务涉及字段。

当码表数据过大，不适合放在 `03-code.sql` 中时，使用 `20+` 编号文件，命名模式为 `codedata_<table>.sql`。

## 菜单和功能编码

SQL 中使用的应用类型映射为：

- `0`：首页（公共）
- `1`：Admin 管理后台
- `4`：用户端
- `7`：开放平台

> 注：`2`（集团端）、`3`（门店端）、`5`（渠道端）、`6`（医院端）为旧项目预留值，当前项目不使用。

菜单编码使用 `MN...` 层级结构，两位一级：

- 一级菜单：`MN01`
- 二级菜单：`MN0101`
- 三级菜单：`MN010101`

功能挂在菜单下面，功能编码前缀取所属菜单编码，并将开头的 `MN` 替换为 `FN`。功能编码同样两位一级，最后两位表示功能动作。

通用功能动作编码含义：

- `00`：菜单查看
- `01`：刷新
- `02`：新建
- `03`：修改
- `04`：删除
- `05`：详情

其他非通用动作从 `10` 开始，按同一菜单下的业务动作顺序递增。新增功能时复用同一菜单下已有编号规律，不要另起一套编码序列。

`cs_function.fn_name` 应只保留动作名称本身，不要附带菜单或分类前缀。例如菜单查看功能统一使用 `菜单查看`，操作按钮仅使用 `新增`、`编辑`、`删除`、`详情` 等动作名。

## 大数据文件

`21-`、`22-`、`23-` 开头的码表数据文件包含大规模静态数据。除非任务明确要求刷新整份数据，否则不要重写、排序或整文件生成。定向修改时，只做最小必要改动，并保留现有 `insert` 格式。

## 编码和换行

多数 SQL 文件包含中文注释和中文值。修改时保留现有文件编码和换行风格。如果 PowerShell 读取 CSV 时出现乱码，应先尝试使用正确的系统默认或历史编码读取，再判断内容和编辑方式。

## 验证要求

完成 SQL 修改前需要验证：

- 运行针对性 diff：`git diff -- <changed files>`。
- 修改表结构时，人工检查对应 `CREATE TABLE` 的逗号、索引、注释和默认值。
- 修改菜单或初始化权限相关数据时，确认 `initialization/` 中相关 `MN...`、`FN...`、应用类型、角色 ID、用户 ID 相互匹配；不要同步 `permission/` 或 `privilege/`。
- 修改字典时，确认字典 ID 区间、表名和字段名符合本文件规则。
- 新增脚本时，确认脚本按预期顺序执行，并在文件头或相邻 README 中说明额外执行顺序要求。
- 发布前优先在一次性数据库或可回滚环境中验证结构和种子数据脚本。
