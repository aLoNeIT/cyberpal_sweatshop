# Hyperf 运行时规范

本文档用于约束 Hyperf 3.1 长驻进程、协程、容器、配置、队列和后台进程的使用方式，避免代码在 HTTP、WebSocket、队列、定时任务等不同运行场景下产生隐式环境依赖。

## 1. 长驻进程与协程

- Hyperf 服务为长驻进程运行，禁止把请求级数据保存到静态属性、单例对象属性或其他长生命周期对象中。
- 用户信息、请求参数、Header、Cookie、文件对象、Session 数据等请求上下文，只能在当前请求链路内使用。
- 跨请求复用的静态缓存只能保存不可变元数据，例如反射结果、注解解析结果、配置映射等；不得保存用户态、权限态或业务可变数据。
- 协程内不得依赖 PHP 全局变量传递业务上下文；确需传递协程上下文时，必须使用 Hyperf 官方上下文能力，并明确生命周期。
- 新增异步、队列、进程、WebSocket 代码时，必须确认其不依赖 HTTP 请求生命周期。

## 2. 容器与依赖注入

- 业务类依赖应优先通过构造函数或 Hyperf 注入方式声明，不得在方法内部反复手动创建复杂依赖。
- 注入到单例或长生命周期对象中的依赖，不得持有请求级状态。
- 控制器 `initialize()` 在对象构造期执行，此时不得读取或调用 `#[Inject]` 注入属性，包括 `request`、`sessionLogic`、`dictLogic`、`container` 等；需要依赖注入对象的逻辑必须放到具体 action、`before*`、`after*` 或其他 action 调用链内执行。`initialize()` 中允许配置不立即求值的闭包，例如 `filter` 中的请求级取值闭包。
- `Controller` 层和 `SessionLogic` 可以访问 HTTP 相关对象；其他层如需当前用户、机构、权限等信息，必须由调用方以普通参数传入。
- 公共 `Logic`、`Model`、`Service`、`Transaction` 不得直接依赖 `RequestInterface`、`SessionInterface`、PSR-7 Request、Response 等 HTTP 对象，`SessionLogic` 除外。
- 新增依赖绑定时必须写入 `config/autoload/dependencies.php`，并确认绑定不会破坏测试和非 HTTP 场景。

## 3. 配置与环境变量

- 业务配置统一放在 `config/autoload/` 下，禁止在业务代码中硬编码环境差异值。
- 数据库、Redis、短信、第三方平台、密钥、Token、证书路径等敏感配置必须通过环境变量或安全配置来源注入，禁止提交真实密钥。
- 新增环境变量时，必须同步更新 `.env.example` 或对应环境说明文档，说明变量用途、默认值和是否必填。
- 配置读取必须提供合理默认值或失败处理；关键配置缺失时应尽早抛出受控异常，避免运行到业务中段才失败。
- 不同环境的配置差异必须通过配置项表达，不得在业务代码中使用主机名、IP、路径等硬编码条件分支。

## 4. 队列、进程与异步任务

- 队列任务、后台进程、定时任务不得读取 HTTP Request、Response、Session、Cookie、Header 等上下文。
- 投递队列时必须传入任务所需的完整业务参数，例如用户 ID、机构 ID、业务主键、操作来源等，不得依赖执行时再从 Session 读取。
- 队列任务必须考虑幂等、重复消费、失败重试和补偿策略；涉及外部调用、文件、短信、支付、状态流转时必须记录可追踪日志。
- 后台任务中发生业务失败时，应返回或记录受控错误；发生非预期异常时必须记录异常上下文并交由统一失败处理流程。
- 单次 HTTP 请求预计超过 10 秒的任务，应优先拆分为异步任务，并提供前端轮询或查询结果的接口。

## 5. 运行时文件与缓存

- 运行时文件、临时文件、导出文件应放入明确的运行目录或临时目录，不得散落在项目源码目录中。
- 临时文件必须有清理策略；涉及敏感数据的临时文件必须限制访问范围并在使用后及时删除。
- 缓存 key 必须包含业务分组和必要的隔离维度，例如应用类型、机构、用户或业务主键，避免跨端、跨机构污染。
- 缓存内容不得保存明文密码、完整 Token、身份证、处方等敏感数据；确需缓存敏感标识时必须脱敏、加密或缩短过期时间。
