## 2. Why —— 为什么做这一段

### 2.1 业务问题
- PRD §7 把「用户管理（列表/详情/禁用/删除）」列为 Admin 后台第一项功能，但只有占位，无边界定义。
- 这是 Admin 价值最直接的体现：**平台运营者必须能看清全平台用户、在异常时干预**。但干预到哪一层、能否看用户会话内容，是个人信息保护法的高风险区，必须显式定义，不能默认"全看"。
- 它直接依赖 07 的 RBAC：删除用户、重置密码需要「用户管理」权限；查看会话内容需要**独立、默认关闭**的权限点。

### 2.2 目标
- G-1：Admin 可检索、查看全平台用户及其用量概览，快速定位异常账户。
- G-2：Admin 可在合规边界内禁用 / 启用 / 删除 / 重置用户密码，处置违规或风险账户。
- G-3：Admin 可全局查看并干预用户的 Agent / Session（停 / 启 / 强制结束），但**不默认拥有查看会话内容的能力**。
- G-4：所有跨用户操作写入 `cs_user_log`，会话内容查看默认关闭、需单独授权且全量审计。

## 3. 范围（In-scope / Non-goals）

### 3.1 In-scope（本期做）
- 用户列表：分页、按状态 / 注册时间 / 用量筛选、按账号 / 昵称搜索。
- 用户详情：基本信息 + 用量概览（token / 估算费用）+ Agent 数 / 活跃 Session 数。
- 禁用 / 启用：`usr_state` 控制，禁用后该 user 无法登录，其 Agent / Session 停止服务。
- 删除：软删除（`usr_delete_time`），处置违规账户。
- 重置密码：生成临时口令或触发重置，写审计。
- 跨用户 Agent / Session 全局查看与干预（停 / 启 / 强制结束）。
- 会话内容查看：作为**独立权限点** `view_user_session_content`，默认不授权，开启后全量审计。

### 3.2 Non-goals（明确不做）
- ❌ **不做用户数据导出 / 批量下载**：避免大规模个人数据落盘，触发合规风险；如需，单独立项。
- ❌ **不做模拟登录（impersonation）**：以用户身份进入其工作区操作——隐私敏感，推到二期+（见 07 §3.2）。
- ❌ **不做用户编辑（改昵称 / 邮箱 / 资料）**：Admin 仅运营处置，不代用户改资料；资料修改权归用户自己。
- ❌ **不做跨用户数据迁移 / 合并**：无组织维度，无此需求。
- ❌ **不做计费扣减**：删除 / 禁用不影响计费逻辑，仅停止服务；扣费不在本期（见 07 §3.2）。
- ❌ **删除时的物理销毁 vs 合规留存策略**：本期软删除保留数据（审计/争议兜底），GDPR「被遗忘权」的物理擦除/匿名化列为 Open Question，不默认执行。

### 4.1 角色（来自 07）
| 角色 | 可操作本模块范围 |
| --- | --- |

| 客服 | 仅查看用户与会话摘要，无写操作 |

### 4.2 用户故事

| US-U1 | 作为运营，我希望按账号搜索并查看用户详情与用量，以便快速定位异常账户 | FR-1, FR-2 |

| US-U6 | 作为超管，我希望在合规授权下查看特定用户的会话内容以处理投诉，以便核实事实 | FR-7 |

## 5. 功能需求详述（FR）

### FR-1 用户列表与检索
- **FR-1.1** 列表数据源：`cs_user WHERE usr_app_type = 4 AND usr_delete_time IS NULL`（排除已软删）。
- **FR-1.2** 支持分页（默认 20/页）、按 `usr_state`（启用/禁用）、注册时间区间、累计用量（token / 费用）排序筛选。
- **FR-1.3** 支持按 `usr_account` / 昵称 模糊搜索。
- **FR-1.4** 列表列：账号、昵称、状态、注册时间、Agent 数、活跃 Session 数、累计 token、累计估算费用、最近活跃时间。

### FR-2 用户详情
- **FR-2.1** 基本信息：`usr_account`、昵称、状态、注册时间、最近登录时间（`usr_login_last`）。
- **FR-2.2** 用量概览：累计 input/output/cache token、累计估算费用（聚合该 user 的计费数据，按 `user_id`）。
- **FR-2.3** 资源概览：Agent 总数、活跃 Session 数、活跃/归档会话数（对照 User 端存储上限，可提示是否触顶）。
- **FR-2.4** 不展示用户明文密码、不展示会话内容（见 FR-7 边界）。

### FR-3 禁用 / 启用
- **FR-3.1** 操作对象 `usr_state`：禁用 → `usr_state = 0`，启用 → `usr_state = 1`。
- **FR-3.2** 禁用后该 user 的 Session 失效（服务端销毁其登录 Session，下次登录被拒）；其活跃 Agent / Session 停止服务（经 gateway 按目标 user 的 session 通知终止）。
- **FR-3.3** 仅持有「用户管理」权限（运营 / 超管）可执行；写 `cs_user_log`（动作：禁用/启用，目标 user_id）。
- **FR-3.4** 不能禁用自己（admin 不能禁用 admin，本模块只操作 `app_type=4`）。

### FR-4 删除（软删除）
- **FR-4.1** 置 `usr_delete_time = NOW()`，不物理删行；列表与登录均按 `usr_delete_time IS NULL` 过滤。
- **FR-4.2** 删除后该 user 无法登录；其 Agent / Session 停止；关联计费数据**保留**（审计 / 争议兜底，见 Non-goals 物理擦除为 Open Question）。
- **FR-4.3** 仅**超管**可删除用户（运营无此权限，见 07 角色集）；写 `cs_user_log`（动作：删除，目标 user_id + 变更前状态）。
- **FR-4.4** 二次确认弹窗（危险操作），前端明确提示「软删除，数据保留但账户不可恢复登录」。

### FR-5 重置密码
- **FR-5.1** 触发后生成临时口令（或发送重置链接），用户下次登录强制改密（复用 User 端首次登录改密机制）。
- **FR-5.2** 仅持有「用户管理」权限可执行；写 `cs_user_log`（动作：重置密码，目标 user_id；**不记录明文口令**）。
- **FR-5.3** 重置不改动 `usr_state`、不触达会话（用户已有 Session 是否失效由产品决策：建议失效以保安全，列为 Open Question）。

### FR-6 跨用户 Agent / Session 全局管理
- **FR-6.1** 列表展示指定 user 的 Agent 与活跃 Session（数据均按 `user_id` 隔离取数）。
- **FR-6.2** 干预动作：
  - 停止 Agent：终止其下所有 Session，标记 Agent 不可用。
  - 强制结束 Session：经 gateway 按**目标 user 的 session** 发终止指令（gateway 已支持 admin service 跨 user session keyed，见 06 设计）。
- **FR-6.3** 干预写 `cs_user_log`（动作：停止 Agent / 结束 Session，目标 user_id + 资源 id）。
- **FR-6.4** 仅运营 / 超管可执行；客服仅查看。

### FR-7 会话内容查看边界（隐私，核心）
- **FR-7.1** 会话内容（对话原文、thinking、tool 调用细节）查看受**独立权限点** `view_user_session_content` 控制，该权限点**默认不授予任何角色**（含运营），仅超管可显式开启并赋给指定角色。
- **FR-7.2** 未持有该权限时，Admin 端仅能看会话**元数据**（session id、时间、token 用量、状态），看不到内容（满足个人信息保护法最小必要原则）。
- **FR-7.3** 持有权限者每次查看会话内容，写 `cs_user_log`（动作：查看会话内容，目标 user_id + session_id + 查看时间）；日志不可篡改。
- **FR-7.4** 查看入口需二次确认：「你将查看用户私人对话内容，该行为会被记录」，明确告知操作人。

## 6. 数据模型变更

### 6.1 结论
**本期零新表。**

| 操作 | 表 | 字段 / 说明 |
| --- | --- | --- |
| 禁用/启用 | `cs_user` | `usr_state`（0/1） |
| 软删除 | `cs_user` | `usr_delete_time` 置时间 |
| 重置密码 | `cs_user` | 覆盖 `usr_pwd` + `usr_salt`，`usr_login_last` 维持 |
| 审计 | `cs_user_log` | `ul_app_type=1, ul_user=admin.id, ul_module/controller/action, ul_extend(JSON: 目标user_id+资源id+变更), ul_remark` |
| 会话内容查看授权 | `cs_function_detail` + `cs_role_permission` | 新增权限点 `view_user_session_content`（`fn_app_type=1`），默认不绑定任何角色 |

> 注：`usr_join_*` 等列仍留空（无租户），不误填。

## 7. 隐私与合规

- **P-1（最小必要）**：Admin 默认仅见会话元数据，会话内容需独立授权（FR-7.1），从根本上限制个人数据暴露面。
- **P-2（告知与审计）**：查看会话内容强制二次确认 + 全量 `cs_user_log` 记录（FR-7.3/7.4），满足可溯源与威慑。
- **P-3（数据留存）**：软删除保留数据用于审计 / 争议，但 GDPR「被遗忘权」的物理擦除 / 匿名化策略未定，列为 Open Question，不默认执行物理销毁。
- **P-4（无批量导出）**：禁止用户数据批量下载（Non-goals），避免大规模个人数据落盘。
- **P-5（不代用户改资料）**：Admin 不编辑用户昵称 / 邮箱 / 资料，资料主权归用户。

## 8. 验收标准

| 编号 | 标准 |
| --- | --- |
| AC-1 | 用户列表仅展示 `app_type=4` 且未软删账户；分页 / 筛选 / 搜索可用 |
| AC-2 | 用户详情展示基本信息 + 用量概览 + Agent/Session 数；不展示明文密码与会话内容 |
| AC-3 | 运营禁用用户后，该 user 登录被拒、其 Session 失效、Agent/Session 停止 |
| AC-4 | 仅超管可删除用户；删除为软删除，列表不再出现，关联计费数据保留 |
| AC-5 | 重置密码后用户下次登录被强制改密；`cs_user_log` 记录动作且不含明文口令 |
| AC-6 | 运营可全局停止某 user 的 Agent / 强制结束其 Session，并经 gateway 实际终止 |
| AC-7 | **默认无角色可查看会话内容**；仅超管显式授权的角色可见，且每次查看写审计 + 二次确认 |
| AC-8 | 所有禁用/启用/删除/重置/干预/查看内容操作均在 `cs_user_log` 留痕，含操作人、目标、资源 |

## 9. 风险与待定项（Open Questions）

| 项 | 说明 |
| --- | --- |
| 删除后物理擦除 vs 合规留存 | GDPR「被遗忘权」要求擦除/匿名化；本期软删除保留。需与法务/合规对齐最终策略 |
| 重置密码是否失效已有 Session | 建议失效以保安全，待定 |
| 会话内容查看的授权粒度 | 当前为角色级全开；是否需要「按次审批 / 时效令牌」更细粒度，待定 |
| 用量概览数据来源 | 取计费聚合表（按 user_id）；需确认聚合表字段与实时性 |
| 批量处置（如批量禁用） | 本期单条操作；批量处置是否需支持，待运营反馈 |
