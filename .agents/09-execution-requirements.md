# 执行要求

本文档用于统一规范执行口径，明确新代码、历史代码和多类规范的同时执行要求。

## 1. 新增代码要求

- 新增代码必须优先遵循本规范。
- 新增代码必须同时满足 `03-coding-standards.md` 中的注释要求，所有公开、受保护、私有、静态、构造、析构及魔术方法都必须补齐 `DocBlock`，类变量、复杂属性也必须补充注释，并确保注释与实现一致。
- 代码提交说明必须使用中文书写，避免使用无意义或过于笼统的提交描述。

## 2. 历史代码要求

- 历史代码在重构、迁移和功能迭代过程中，逐步向本规范收敛。

## 3. 综合执行要求

- 目录规范、分支规范、数据库规范和编码规范应同时执行，不能只执行其中一部分。
- 每次提交前必须先对本次变更执行代码审计，至少检查以下内容：
  - 审查流程、优先级和检查维度必须参考 `17-code-review-standards.md`；
  - 代码规范性是否符合项目规范与分层约定；
  - 方法注释与关键注释是否完整，且与实际行为一致；
  - 语法是否有效，新增或修改文件不得存在语法错误；
  - 新增或修改接口是否同步检查权限注解、白名单、登录态、数据范围和响应脱敏；
  - 新增或修改配置是否同步环境变量说明，且未提交真实密钥、Token、账号密码等敏感信息；
  - 新增或修改异常处理、日志、队列、后台进程时，是否符合运行时、异常日志和安全规范；
  - 新增或修改接口时，是否同步更新测试用例，或明确说明暂不更新原因。
- 提交前审计未通过时，禁止执行 `push`，必须先修复问题后再提交和推送。
- 本地 `commit` 成功后，应及时推送到远程仓库或远程服务器，避免出现仅本地留存、远端缺失的提交。
