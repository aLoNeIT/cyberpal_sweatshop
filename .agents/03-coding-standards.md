# 编码规范

本文档用于统一项目中的编码习惯、命名规则、注释规则、返回结构和常见基础设施使用约定。

## 1. 通用规范

- 所有方法必须编写注释，包括 `public`、`protected`、`private`、`static` 方法，以及自定义行为的构造方法、析构方法和魔术方法。
- 关键代码必须补充必要注释。
- 文件名必须与类名保持一致，并按职责追加对应类型后缀。
- `use` 引用需要优先使用 `as` 进行类型重命名，避免歧义。
- 新增控制器时，必须优先保证继承链路中最终包含 `DictCrudController`，利用其标准 CRUD 模板快速开发。详见 [项目分层规范 - DictCrudController 标准 CRUD 开发模式](05-layered-architecture.md#11-dictcrudcontroller-标准-crud-开发模式)。
- 控制器在需要请求对象时，优先沿用继承链中已声明且已完成注入或赋值的 `request` 成员对象，并直接使用 `$this->request` 读取 query、body、file 等数据。
- 仅当继承链中不存在可直接使用的 `request` 成员对象时，子类控制器才允许自行通过依赖注入声明 `RequestInterface $request`。
- 控制器对外接口返回值一般情况下必须为 `JsonTable` 对象；返回图片、文件流等二进制内容时，允许返回框架约定的特定 `Response` 对象。
- 控制器 `public` 方法返回值类型声明为 `JsonTable|Psr\Http\Message\ResponseInterface`，明确方法在正常返回与文件流返回两种场景下的联合类型。
- 调用返回 `JsonTable` 对象的方法后，若还有后续业务逻辑需要处理，应使用 `Helper::throwifJError()` 提前终止失败流程；若直接将其作为控制器最终返回值，则无需包装，直接 `return` 即可，由框架统一处理 JsonTable 对象（无论成功或失败）。
- 对数组进行为空判定时，统一使用 `empty()` 方法，禁止使用 `=== []` 等严格比较形式。
- 接口或方法允许返回空数组（`[]`）时，调用方无需对空数组进行额外的为空判断或错误处理；空数组视为正常业务结果。
- 控制器中的 `public` 方法视为对外接口函数，命名统一使用 `snake_case` 风格，方法名最后一个单词必须使用 `index`、`save`、`update`、`delete`、`read` 标记接口所属的 RESTful 类型，例如 `UserController::password_update` 表示用户密码更新接口。
- 控制器子类继承父类已有接口方法且没有业务行为变化时，禁止仅为补充注释、权限注解或调整方法签名而重写空包装方法；应复用父类实现，确需差异化权限码时应优先调整公共权限映射或父类配置规则。
- 控制器 `initialize()` 仅允许设置字典 ID、事务开关、排序、静态过滤规则等构造期基础配置；禁止在 `initialize()` 中使用 `#[Inject]` 注入对象或读取请求上下文。若 `filter` 等配置需要请求级数据，应使用闭包延迟到 action 执行链路内求值。
- 子类对父类已声明的全局配置变量赋值时，统一在 `initialize()` 方法中执行，例如 `dictId`、`transaction`、`filter`、`order`、`excludePrefix` 等；除非该变量需要通过 `#[Inject]` 或构造注入等依赖注入机制实例化，否则禁止通过子类属性默认值覆盖父类配置。
- 控制器中的 `protected`、`private` 方法仅供类内部复用，命名统一使用 `camelCase` 风格，不得作为路由对外暴露。
- 只有 `Controller` 层和 `SessionLogic` 可以直接访问 HTTP 相关对象或上下文方法，例如 `request`、`session`、上传文件、Header、Cookie 等。
- 除 `Controller` 层和 `SessionLogic` 外，其他代码不得直接读取或依赖 HTTP 上下文；需要的数据必须由控制器解析、过滤、校验后作为普通参数传入，避免在后台进程、WebSocket、定时任务等非 HTTP 场景中失效。
- 禁止为了单次调用、无复用价值、无语义增益且内部逻辑非常简单的语句额外封装 `protected` 或 `private` 方法，例如仅返回 `time()`、仅调用 `$model->save()`、仅转发事务方法、仅封装一条简单查询或仅转发一个全局函数/对象方法；此类逻辑原则上应直接内联。
- 仅当封装能够表达稳定业务概念、承载复杂分支、消除实际重复、隔离外部依赖或明确事务/锁等副作用边界时，才允许新增辅助方法，并应通过方法名和注释说明封装意图。
- `SessionLogic` 派生类覆写 `buildSessionData()` 时，应在该函数内集中完成本端登录会话数据处理、校验和字段补充；若某段逻辑没有复用场景，且内联后当前函数不超过 100 行，不应额外拆成 `assert*`、`*Available`、`*ById`、`sessionData` 等只被该函数调用一次的辅助函数。
- 父类属性已经声明且注入或赋值有效值时，子类禁止重复声明同名属性，避免覆盖、歧义或重复维护。
- 子类因模型元数据、配置覆盖等场景必须重写父类已有属性时，必须在属性上方补充 `/** @inheritDoc */` 注释，便于 IDE 和静态分析识别继承关系。

## 2. 后缀规范

- 控制器类使用 `Controller` 后缀。
- 模型类使用 `Model` 后缀。
- 逻辑类使用 `Logic` 后缀。
- 常量容器类使用 `Const` 后缀。
- 业务常量统一放在 `app/common/constants` 目录下，并按职责拆分为 `XxxConst` 类；禁止在 `Controller`、`Logic`、`Model` 等业务类中直接声明可复用业务常量。仅允许类内私有、不可复用的实现细节常量保留在当前类中。
- 异常类使用 `Exception` 后缀。
- `Facade` 类使用 `Facade` 后缀。
- 契约或接口定义放在 `contract` 目录下，并使用 `Contract` 后缀。
- 监听器类使用 `Listener` 后缀。
- 进程类使用 `Process` 后缀。
- 事务类使用 `Transaction` 后缀。
- 同一职责下的基础类也必须保留所在层后缀，例如 `BaseController`、`BaseModel`、`BaseLogic`、`BaseTransaction`。
- 通用核心基类可使用 `CoreModel`、`CoreLogic` 这类命名，但仍需保留所属职责后缀。

## 3. PHP 规范

- 遵循 PSR 规范：
  - `PSR-1`
  - `PSR-2`
  - `PSR-12`
- 参考地址：
  - `https://learnku.com/docs/psr`
  - `https://hyperf.wiki/3.1/#/`
- 代码应尽量使用强类型约束。
- 常量值、字面量与变量进行比较时，常量值或字面量必须位于左侧，变量位于右侧，例如 `0 === $state`，禁止写成 `$state === 0`，避免少写一个 `=` 时将比较表达式误写为赋值表达式。

## 4. PHP 注释规范

### 4.1 总体要求

- 所有方法必须在定义前编写 `DocBlock` 注释，不允许遗漏。
- 类、`trait`、`interface`、`enum`、业务常量、复杂属性也必须补充注释。
- 模型类必须补充类级注释，说明模型对应的业务实体。
- 注释内容必须与代码行为保持一致；修改方法签名、返回值、异常或副作用时，必须同步更新注释。
- 注释应优先说明“做什么、为什么这样做、输入输出约束、失败条件、副作用”，禁止只重复代码字面含义。
- 子类重写父类已有方法、属性或变量时，如继承父类注释语义即可表达当前行为，统一使用 `/** @inheritDoc */`，不需要重复编写单独注释。

### 4.2 方法注释要求

- 每个方法注释至少包含：方法作用、参数说明、返回值说明。
- 存在业务前置条件、状态流转、事务、锁、幂等控制、第三方调用、消息投递、文件操作时，必须在注释中明确说明。
- 可能抛出业务异常、参数异常、第三方异常时，必须补充 `@throws` 说明。
- 参数或返回值为复杂数组时，必须在注释中说明关键字段含义；可表达为结构化数组说明时，不得省略。

推荐模板：

```php
/**
 * 提交处方审核
 *
 * 业务规则：
 * - 仅待审核状态允许提交
 * - 提交成功后写入审核日志并投递异步通知
 *
 * @param int $prescriptionId 处方ID
 * @param int $operatorId 操作人ID
 * @return JsonTable
 * @throws BusinessException 当处方状态不允许提交时抛出
 */
public function submitAudit(int $prescriptionId, int $operatorId): JsonTable
{
}
```

### 4.3 行内注释要求

- 复杂分支、兼容逻辑、性能优化、事务边界、锁控制、历史遗留处理必须补充行内或块注释。
- 行内注释应解释业务原因、兼容背景或边界条件，不得写成“定义变量”“循环数组”这类无意义描述。
- 已废弃逻辑不得长期以注释代码形式保留，历史信息统一通过 `git` 追溯。

### 4.4 分层注释要求

- `Controller`：注释中必须体现接口用途、请求方式、核心参数、返回结构、登录或权限要求。
- `Logic`：注释中必须体现业务规则、前置条件、状态流转、事务与副作用。
- `Model`：注释中必须体现模型对应实体含义；`scope` 或查询方法需说明筛选条件与返回的查询构造器用途。
- `Service`：注释中必须体现外部依赖、调用约束、失败处理策略和返回结果语义。
- `Test`：测试方法名无法完整表达意图时，必须注释测试目的、前置条件、样本依赖和断言重点。

### 4.5 禁止事项

- 禁止缺失方法注释。
- 禁止注释与实际代码行为不一致。
- 禁止使用“定义变量”“判断是否为空”“调用方法”这类无信息量注释。
- 禁止把业务规则只写在注释中而不落到代码校验中。
- 禁止用大段注释掩盖职责混乱、命名不清或可拆分的坏代码。

## 5. Response 返回结构规范

统一返回结构由 `JsonTable` 对象承载，包含以下字段：

- `state`
  - 状态码
  - `0` 表示成功，非 `0` 表示失败
  - 状态码需统一管理
- `msg`
  - 消息内容
  - 当列表页查询时，`msg` 中包含分页信息：
    - `curr`：当前页码
    - `page`：总页数
    - `num`：每页数据量
    - `count`：总数据量
- `data`
  - 其他扩展数据体

返回数据的 `key` 一律使用小写下划线风格。

### 5.1 JsonTable 安全获取规范

- `Helper::throwifJError($jsonTable)`：用于获取 JsonTable 后还需继续执行业务逻辑的场景。成功时返回该 JsonTable 对象供后续使用；失败时自动抛出业务异常，由框架统一异常处理器转换为错误响应。
- 当 JsonTable 对象直接作为控制器最终返回值时，无需使用 `throwifJError` 包装，直接 `return` 即可，框架会统一渲染 JsonTable 对象（无论成功或失败状态）。
- 禁止手动调用 `$jsonTable->isSuccess()` 后再分叉处理，避免遗漏异常传播和错误码丢失。
- 控制器层的异常已统一由框架处理器接管，业务代码中无需在控制器内捕获业务异常。

## 6. 数据校验规范

- 必须进行传参校验。
- 必须进行文件上传校验。

## 7. Redis 规范

- 缓存名称按分组设计，使用 `:` 分割。
- Redis key 推荐使用 `{模块}:{功能}:{维度1}:{维度2}` 格式，新增 key 常量时必须确认没有与已有常量值重复。
- 集合数据存储优先考虑 `hash` 结构。
- 队列处理必须考虑失败场景。
- 建议队列信息中记录创建时间、失败次数等字段。
- 分布式锁必须设置过期时间，并在 `finally` 中释放；锁值应使用随机 token，释放前先比对 token，释放失败必须记录 warning 日志。
- 编码生成 Redis 序号统一使用 `code:create:seq:{prefix}:{scope}`，初始化锁统一使用 `code:create:seq:lock:{prefix}:{scope}`。
- 所有业务编码生成必须通过 `GenerateCodeLogic` 统一入口。

## 8. 路由规范

- 使用 RESTful 设计风格。
- URI 中应包含版本号；不包含版本号的默认为最新版本。
- 对外暴露的路由 URI 如果包含组合词，必须使用小写下划线分割。
- 内部函数如果直接暴露为 URI，对外名称使用下划线风格。
- 内部调用函数使用驼峰命名。
- 中间件优先配置在路由分组上，分组内控制器默认继承中间件约束，避免在控制器中重复单独配置。
- 新增或修改接口时，应同步维护对应的接口文档。

## 9. Facade 与容器规范

- 单例模式优先使用容器。
- 需要静态调用时可使用 `facade`。
- 类似 TCP 连接的类库，如果使用 `facade` 模式，必须设置为总是创建新对象。
