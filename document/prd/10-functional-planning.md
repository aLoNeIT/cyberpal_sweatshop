## 4. 通知通道设计（新模块 · 闭环 R-015）

> 审计红色缺失项之一。R-015 用量预警、以及未来的账户安全（密码重置、禁用/启用）、运营通知都依赖统一通道。

### 4.1 Why
- 01 §3.4 把 R-015 列为 P2，但**无任何通知基建**——预警触发后无处投递。这是功能不可闭环的根因。
- 统一通道避免各功能各自调邮件/短信，便于审计、退订与渠道扩展。

### 4.2 范围
- **In-scope**：
  - 渠道：站内信（MVP 必需）、邮件（MVP 必需，SMTP/SES）、Webhook（Later 预留）。
  - 事件：用量预警（R-015）、密码重置通知、账户禁用/启用通知、会话异常终止（可选）。
  - 投递：异步队列、失败重试、聚合沉默（同事件 N 分钟内不重复）。
- **Non-goals**：
  - ❌ 不含营销推送 / 广告。
  - ❌ 不含实时聊天消息（那是会话本身，非通知）。
  - ❌ 不含短信（本期无短信网关）。

### 4.3 用户故事
| 编号 | 故事 |
|------|------|
| US-N1 | 作为用户，我希望用量接近阈值时收到提醒，以便及时清理或升配 |
| US-N2 | 作为用户，我希望密码被重置时收到通知，以便察觉异常 |
| US-N3 | 作为超管，我希望系统事件可投递到统一通道，以便运营不各自接邮件 |

### 4.4 关键 FR
- **FR-N1** 通知模板：`cs_notify_template`（code、channel、title、body 模板、变量占位）。
- **FR-N2** 站内信：`cs_notify_inbox`（user_id、template_id、payload、read_at、created_at）；前端设置页「消息中心」读取。
- **FR-N3** 发送日志：`cs_notify_log`（user_id、channel、status、retry、error、sent_at）；用于审计与送达率统计（指标体系健康指标）。
- **FR-N4** 异步投递：事件 → 消息队列 → 发送 worker；失败指数退避（最多 3 次）。
- **FR-N5** 聚合沉默：同一 `(user_id, template_code)` 在 10 分钟内仅投递 1 次（防轰炸）。
- **FR-N6** 邮件渠道：配置 SMTP/`cs_system_config` 存凭据（加密），默认关闭，超管开启。
- **FR-N7** R-015 接入：计费聚合 Job 检测到用户用量 ≥ 阈值 → 发 `usage_warning` 通知（站内信 + 邮件）。

### 4.5 数据模型影响
- **需新建**：`cs_notify_template`、`cs_notify_inbox`、`cs_notify_log`（3 张，均为 `cs_` 前缀，符合表命名规定）。
- 复用：`cs_user`（收件人）、`cs_system_config`（邮件凭据）。

### 4.6 风险 / 待定
| 项 | 说明 |
|----|------|
| 邮件凭据安全 | `cs_system_config` 存密文；或走环境变量注入，不落库 |
| 送达率基线 | 需先有数据，指标 §5 健康项「通知送达率」目标待定 |
| Webhook 签名 | Later，需 HMAC 校验 |

## 5. 指标体系（新模块 · 北极星 / 驱动 / 健康）

> 审计红色缺失项之二。当前无任何指标定义，无法度量「产品在帮用户创造价值」与否。

### 5.1 北极星指标（North Star）
**WAAS — 周活跃 Agent 会话数（Weekly Active Agent Sessions）**
- 定义：自然周内，发生过 ≥1 轮有效对话（含用户消息 + Agent 响应）的会话去重计数。
- 为什么：直接反映「用户在用 Agent 创造/完成工作」，兼顾活跃与价值，不被单纯注册数误导。
- 数据源：`cs_sessions`（status=active/archived）+ `cs_messages`（有 user+assistant 消息对），按周聚合。
- 基线：上线后首月实测；目标：环比 +X%（首版不设硬 KPI，先观测）。

### 5.2 驱动指标（Inputs / Drivers）
| 维度 | 指标 | 定义 / 数据源 | 目标方向 |
|------|------|------|------|
| 获取 | 注册转化率 | 访问注册页 → 成功注册 / `cs_user`(app_type=4) | ↑ |
| 激活 | 首 Agent 创建率 | 新用户 7 日内创建 ≥1 Agent 比例 | ↑ |
| 激活 | 会话发起率 | 有 Agent 的用户中发起会话比例 | ↑ |
| 参与 | 续聊/分叉率 | 历史会话中续聊或分叉的比例 | ↑ |
| 参与 | Skill/MCP 挂载率 | 有 ≥1 skill 或 mcp 挂载的 Agent 比例 | ↑ |
| 留存 | D7 / D30 留存 | 注册后第 7/30 日仍有会话的用户比例 | ↑ |
| 经营(Admin) | 平台活跃用户数 | 周内有会话的 `cs_user`(app_type=4) 数 | ↑ |
| 经营(Admin) | 平台总 Agent / 总 token | 全平台累计 / `cs_agents` / `cs_billing_records` | 观测 |

### 5.3 健康指标（Guardrails / Health）
| 指标 | 定义 / 数据源 | 预警线 |
|------|------|------|
| 会话错误率 | SSE 失败 / 总会话（`cs_events` type=error） | > 2% 告警 |
| P95 首 token 延迟 | Agent 响应首 token 耗时（gateway 埋点） | > 8s 告警 |
| 归档任务成功率 | 自动归档成功 / 触发（`ArchiveExpiredSessionsCommand`） | < 99% 告警 |
| 存储上限触顶率 | 触顶拒绝创建的用户占比 | 观测，高则调上限 |
| 通知送达率 | `cs_notify_log` sent / total | < 95% 告警 |
| Agent 启动失败率 | pi-agent 进程启动失败 / 总启动 | > 1% 告警 |

### 5.4 指标落地（Non-goals / 范围）
- ❌ 本期**只定义指标体系 + 采集埋点 + Admin 经营看板雏形**；不做复杂 BI、不做告警自动化（告警仅在指标越线时记录，人工介入）。
- 采集：事件总线 / 中间件埋点 → `cs_metrics_*` 或复用 `cs_events`；具体存储方案为 Open Question。
- 看板：Admin 端「经营看板」复用 09 计费大盘的聚合能力扩展。

### 5.5 风险 / 待定
| 项 | 说明 |
|----|------|
| 指标存储 | `cs_metrics_daily` 等聚合表待建，或走时序库；待架构拍板 |
| 北极星合理性 | WAAS 是否最佳北极星，需首版数据验证后复评 |
| 隐私 | 指标聚合须满足最小必要，不暴露个人会话内容（延续 08 §7） |
