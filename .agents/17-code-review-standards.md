# 代码审查标准

本文档用于统一代码审查触发时机、问题分级、检查维度、审查流程和项目特定检查项。

## 1. 审查触发时机

| 时机 | 触发条件 | 审查范围 |
|------|----------|----------|
| PR 提交时 | 任一 PR 指向 `develop` 或 `master` | 全量 diff |
| 每日定时 | 工作日定时检查 | 昨日至今所有未审查变更 |
| 紧急修复 | `hotfix` 分支合并 | 仅变更文件，加速审查 |
| 重构完成后 | 单次变更超过 200 行 | 深度审查与架构评审 |

## 2. 问题优先级

### 2.1 P0 阻断项

P0 问题必须修复后才能合并：

- 安全漏洞，例如 SQL 注入、XSS、未授权访问、敏感信息泄露。
- 数据风险，例如数据丢失、数据损坏、并发写冲突。
- 线上事故风险，例如可能直接导致 500 错误或服务不可用。
- 破坏性变更，例如修改 API 契约、常量值、数据库结构但未同步调用方、文档、测试或迁移。

### 2.2 P1 重要项

P1 问题原则上应在本次变更中修复：

- 请求参数、外部数据源等输入校验缺失或不足。
- 死代码、从未执行的分支、不再调用的方法。
- 性能隐患，例如 N+1 查询、未索引的 `LIKE` 查询、大循环中重复查库。
- 异常处理不完整，例如 `catch` 后吞掉异常、未记录日志。
- 资源泄漏，例如文件句柄、Redis 连接、分布式锁未释放。

### 2.3 P2 优化项

P2 问题可按风险放入下一迭代处理：

- 代码注释与文档缺口。
- 变量或方法命名不够自解释。
- 重复代码可提取为公共方法。
- 测试覆盖不足的边界条件。
- 日志级别不合理。

## 3. 审查维度

### 3.1 安全性

- 用户输入入库或参与查询前是否使用参数绑定、白名单过滤或项目统一查询封装。
- 是否存在直接拼接 SQL、滥用 `DB::raw` 或将未过滤字段用于排序、查询、更新、删除。
- 手机号、身份证、Token、密钥等敏感信息是否在日志和响应中脱敏。
- 文件上传是否校验类型、大小、路径和访问范围。
- Redis key 是否存在命名碰撞、跨模块复用或租户隔离缺失风险。

### 3.2 正确性

- 常量、枚举、方法签名或 API 契约变更后，所有调用方、测试和接口文档是否同步更新。
- 新增字段是否有数据库迁移、模型字段、字典配置和文档同步。
- 分支条件是否覆盖 `null`、空字符串、`0`、空数组等边界。
- 重构前后的业务行为是否等价，必要时应对照旧调用链验证。
- 并发场景是否存在重复生成、重复提交、状态覆盖或锁失效风险。

### 3.3 可维护性

- 类和方法是否保持单一职责，复杂逻辑是否沉淀到合适层级。
- 魔法数字、魔法字符串是否提取为常量或已有配置。
- 调用链是否过长，超过三层简单委托时应审视是否过度封装。
- 废弃方法是否已全局确认无调用方并清理。
- 新增 Model 局部作用域是否遵循项目命名和返回类型规范。

### 3.4 性能

- 是否存在循环内数据库查询、重复 Redis 调用或可批量处理却逐条处理的代码。
- `LIKE 'prefix%'`、`orderBy`、关联过滤字段是否有合适索引支撑。
- Redis key 是否设置合理过期时间，永久 key 是否有明确业务原因。
- 分布式锁持有时长、等待窗口和降级策略是否合理。
- 长事务中是否包含慢查询、外部接口、文件操作或队列副作用。

### 3.5 测试

- 核心业务方法、异常路径、并发或锁边界是否有测试覆盖。
- 测试数据是否覆盖边界值和历史兼容值。
- Mock/Stub 是否真实反映被替换组件的关键行为。
- 断言是否验证具体返回值、状态和副作用，禁止空壳断言。
- 行为变更必须先补充能失败的回归测试，再修改生产代码。

## 4. 审查流程

1. 提交者在 PR 或提交说明中写清楚变更目的、影响范围、测试方式和需要重点审查的区域。
2. 自动检查先执行语法检查、静态分析和自动化测试。
3. 审查人按安全、正确性、可维护性、性能、测试维度一次性给出完整反馈，并标注 P0/P1/P2。
4. 提交者按优先级处理反馈；P0 必须修复，P1 原则上本次修复，P2 可记录后续任务。
5. 修复后重新执行相关测试和必要的全量验证，再进入复审。

## 5. 审查职责

### 5.1 审查人职责

- 一次审查尽量给出完整反馈，避免反复追加同类问题。
- 每条意见说明问题、风险、建议方向和优先级。
- 对外部审查意见应结合当前代码库验证，不盲目套用通用建议。
- 对可选优化和必须修复的问题明确区分，避免阻塞无风险改进。

### 5.2 提交者职责

- 提交前先自查 diff，减少低级错误和无关变更。
- 单次 PR 尽量控制在 500 行以内，超过时优先拆分。
- 对审查意见逐条处理或给出技术理由。
- 修改后同步测试、接口文档、项目规范或迁移文件。

## 6. 项目特定检查项

### 6.1 编码生成

- 所有业务编码生成必须通过 `GenerateCodeLogic` 统一入口。
- 禁止在业务 Logic 中直接调用 `Db::table()` 做前缀查询、自增或随机重试。
- 新增编码类型时，应在 `GenerateCodeLogic` 中新增方法，并在对应 Model 中补充 `scopeLastCodeByPrefix`。

### 6.2 Redis key 与锁

- Redis key 使用 `{模块}:{功能}:{维度1}:{维度2}` 格式。
- 新增 Redis key 常量时必须确认没有与已有常量重复。
- 编码生成序号 key 示例：`code:create:seq:{prefix}:{scope}`。
- 编码生成序号初始化锁 key 示例：`code:create:seq:lock:{prefix}:{scope}`。
- 分布式锁必须有过期时间，必须在 `finally` 中释放。
- 锁值应使用随机 token，释放前必须比对 token，释放失败应记录 warning 日志。

### 6.3 异常处理

- `catch (Throwable $throwable)` 不允许空处理。
- 可预期业务失败返回受控 `JsonTable`，非预期异常必须记录日志并返回统一错误。
- 日志中不得输出完整 Token、密码、验证码、身份证、银行卡、第三方密钥等敏感信息。

### 6.4 Model 局部作用域

- 业务代码不得为图省事直接 `Db::table()` 查询常规模型数据。
- 查询条件优先沉淀为 Model `scopeXXX`，Logic 层只传业务参数并执行终结操作。
- Model 类文档块应补充 `@method static Builder xxx(...)`，方便 IDE 和静态分析识别。

### 6.5 字段丢弃

- `unset()`、白名单过滤或忽略调用方传入字段时，必须在注释、接口文档或参数校验中说明原因。
- 字段仅用于兼容旧客户端、仅用于计算、不持久化或禁止前端修改时，应在对应钩子附近写清楚。

## 7. 模板

### 7.1 PR 描述模板

```markdown
## 变更说明

## 变更类型
- [ ] 新功能
- [ ] Bug 修复
- [ ] 重构
- [ ] 性能优化
- [ ] 测试补充

## 测试

## 影响范围

## 审查要点
```

### 7.2 审查结论模板

```markdown
## 审查总结

| 类别 | 数量 | 说明 |
|------|------|------|
| P0 阻断 | 0 | 无 |
| P1 重要 | 0 | 无 |
| P2 优化 | 0 | 无 |

## 结论

[通过合并 / 修复后合并 / 需要重新设计]
```
