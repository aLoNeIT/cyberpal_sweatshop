# 异常与日志规范

本文档用于统一业务异常、错误码、日志记录、审计追踪和对外错误响应要求。

## 1. 异常与错误返回

- 可预期的业务失败应优先返回统一 `JsonTable` 错误结果，或抛出受控业务异常，由统一异常处理器转换为 `JsonTable` 响应。
- 不可预期异常不得把堆栈、SQL、文件路径、密钥、内部类名等调试信息直接返回给前端。
- Controller 对外接口一般返回 `JsonTable`；图片、文件流等二进制场景可返回框架约定的 `Response` 对象。
- Logic 层返回给 Controller 的业务结果应保持 `JsonTable` 结构一致，不得混用数组、字符串、布尔值表示业务失败。
- 捕获异常后不得静默吞掉；必须返回受控错误或记录日志后继续抛出。

## 2. 错误码管理

- 业务错误码必须统一维护，禁止在各处随意新增无说明的魔法数字。
- 新增错误码时必须说明错误含义、适用场景和是否可直接展示给前端。
- 同一类业务失败应复用已有错误码，避免同义错误码重复扩散。
- 参数错误、权限错误、登录失效、数据不存在、状态不允许、外部服务失败等通用错误应使用固定错误码。
- 对外错误消息应清晰但不过度暴露内部实现；内部排查所需细节放入日志。

## 3. 日志记录

- 日志必须服务于排查问题和审计追踪，禁止记录无意义的大量调试噪声。
- 异常日志至少包含异常类型、错误消息、文件行号、关键业务主键和调用场景。
- 不得在日志中明文记录密码、盐值、完整 Token、验证码、身份证、银行卡、处方明细、支付凭据、第三方密钥等敏感信息。
- 记录请求参数时必须先过滤敏感字段；文件内容、二进制内容、大数组和大文本不得直接完整写入日志。
- 重要外部调用应记录请求摘要、响应摘要、耗时、重试次数和业务主键，便于定位链路问题。

## 4. 审计与追踪

- 敏感操作应记录审计信息，包括操作人、应用类型、机构范围、业务主键、操作动作、操作前后关键状态和操作时间。
- 同一请求链路应尽量携带 request id 或 trace id；跨队列、进程、外部调用时应将链路标识作为普通参数继续传递。
- 后台任务、队列任务、定时任务的日志必须包含任务类型和任务参数摘要，避免失败后无法定位来源。
- 审计日志和排查日志职责不同；审计日志关注“谁在什么时候做了什么”，排查日志关注“系统为什么失败”。

## 5. 日志级别

- `debug` 仅用于本地开发或临时排查，禁止长期输出高频业务数据。
- `info` 用于记录关键业务节点、任务开始结束、外部调用摘要等正常流程信息。
- `warning` 用于记录可恢复但需要关注的问题，例如重试、降级、数据不完整。
- `error` 用于记录业务无法继续、外部服务失败、非预期异常等错误。
- `critical` 用于记录影响核心链路或可能造成数据不一致的严重问题。
