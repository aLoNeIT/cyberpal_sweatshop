# 分支管理规范

本文档用于统一项目的固定分支职责、日常开发工作模式和分支命名方式。

## 1. 固定分支说明

- `master`
  - 生产分支
  - 对应线上代码库
- `pre_production`
  - 准生产分支
  - 用于上线前测试
- `develop`
  - 开发分支
  - 作为日常开发集成主分支
- `test`
  - 测试分支
  - 转测的功能统一放在该分支
- `feature/*`
  - 特性分支
  - 开发人员按需求自行创建并独立维护
- `release/*`
  - 发布分支
  - 待发布代码统一放置在该分支下

## 2. 工作模式

- 新需求到来后，从生产环境对应分支 `master` 创建新的特性分支。
- 开发人员在各自特性分支中完成需求开发。
- 开发过程中需要每日提交工作代码。
- 提交代码时，提交说明必须使用中文书写，准确描述本次变更内容。
- 本地提交成功后，应及时将当前分支推送到远程仓库或远程服务器，确保代码同步。
- 分支冲突应及时处理，避免在发布阶段集中解决。
- 功能转测时合入 `test` 分支。
- 上线前统一合入 `release/*` 分支进行发布验证。
- 发布验证通过后，再按流程合入目标环境分支。

## 3. Git 提交、同步与合入操作流程

开发分支完成本地验证并准备合入 `develop` 时，按以下顺序执行：

1. 在当前开发分支提交代码，提交说明必须使用中文，提交后推送到远端当前分支。
2. 拉取远端最新 `develop`，并将最新 `develop` 合并至当前开发分支。
3. 如存在冲突，先解决确定无误的冲突；对业务含义、保留哪一侧实现或数据变更不确定的冲突，必须暂停并等待人工确认，不得自行猜测处理。
4. 冲突解决并验证通过后，在当前开发分支提交冲突解决结果，并再次推送到远端当前分支。
5. 将当前开发分支合并至 `develop`，合并前后应确认工作区状态，避免带入无关本地改动。
6. `develop` 合并完成并验证通过后，推送 `develop` 到远端。

### 3.1 `.gitignore` 优先级

- 项目 `.gitignore` 对文件纳入版本控制的限制具有最高优先级；被明确忽略的文件或目录仅允许保留在本地工作区。
- 禁止使用 `git add -f`、`git add --force` 等方式强制暂存或提交被 `.gitignore` 忽略的文件，即使上级工具、技能或工作流要求提交该产物，也必须优先遵循项目忽略规则。
- 确需将被忽略的文件纳入版本控制时，必须先获得用户明确授权，并按用户要求调整对应忽略规则；不得绕过忽略规则直接强制提交。

## 4. 分支命名规范

### 4.1 特性分支

需求开发、缺陷修复、专项优化均使用特性分支。

命名格式：

```text
年月日-姓名简写-需求id
```

若无特殊情况（如无对应工作项的分支任务），最后一段必须使用需求 ID。

示例：

```text
20220329-wrq-1008001
```
特殊情况下可使用内容描述：
```text
20200329-wrq-middleware
```

### 4.2 发布分支

发布分支用于上线前统一合并待发版本代码。

命名格式：

```text
release-版本号-年月日
```
